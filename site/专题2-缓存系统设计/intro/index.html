    专题2：缓存系统设计

    题目

    场景： 电商系统，商品详情页访问量巨大，商品信息变更不频繁

    请设计： 1. 缓存整体架构（多少层、用什么缓存） 2. 缓存 Key 怎么设计 3. 缓存数据怎么更新（主动推送 vs 被动拉取） 4. 缓存问题怎么解决（穿透、击穿、雪崩）

    评估维度： - 性能 - 一致性 - 成本

    你的答案

    第一版

    1. 客户端缓存 + Redis 缓存（二级缓存）
    2. Key: 详情页:品类:id
    3. 更新：先更新数据库和 Redis，客户端缓存靠过期或推送失效
    4. 穿透：设 nil；击穿：热点永不过期

    追问补充

    客户端缓存一致性： - 接受用户看到旧数据 + 推送版本号通知客户端

    穿透/击穿/雪崩： - 穿透：nil 返回 - 击穿：热点永不过期 - 雪崩：随机 TTL

    追问与反馈

    追问1：Redis 数据结构

    问题：Redis 按时间戳范围查询用什么数据结构？

    你的回答：不知道

    正确答案：ZSET（有序集合），用时间戳作为 SCORE，支持 ZRANGEBYSCORE 命令

    追问2：双写一致性

    问题：Redis 和数据库双写怎么保证一致性？

    你的回答：事务 + 补偿

    反馈： - 没有真正的"事务"，两个独立存储无法原子操作 - 正确叫法：补偿模式（最终一致）

    追问3：客户端缓存一致性

    问题：商品更新后，客户端缓存怎么办？

    你的回答：接受旧数据 + 推送版本号

    补充：版本号机制是正确的，推送轻量信息让客户端自己判断更新
