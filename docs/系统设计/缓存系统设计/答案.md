# 缓存系统设计 - 参考答案

## 整体架构

```
请求
 ↓
L1 本地缓存（Caffeine，热点 1 万 SKU，TTL 5s）
 ↓ miss
L2 Redis 集群（全量，TTL 1h）
 ↓ miss
数据库（MySQL）
 ↑
异步回写 Redis + 更新 L1
```

## 三大问题解决方案

### 缓存穿透
**布隆过滤器**：启动时加载全量 SKU ID 到 Bloom Filter
- 请求先过滤：不在 Filter 中的直接返回 404
- 新增 SKU 实时更新 Bloom Filter

### 缓存击穿
**互斥锁 + 逻辑过期**：
```
获取缓存 → 命中（含逻辑过期时间）
        ↓ 未过期：直接返回
        ↓ 已过期：尝试获取分布式锁
                  ↓ 获取成功：异步刷新缓存，返回旧值
                  ↓ 获取失败：返回旧值（短暂不一致可接受）
```

### 缓存雪崩
- TTL 加随机偏移：`TTL = 基础时间 + random(0, 300s)`
- 多级缓存兜底：L1 不受 Redis 雪崩影响
- Redis Cluster 保证单点故障不影响整体

## 数据一致性策略

采用 **Cache Aside + 延迟双删**：
1. 更新 DB
2. 删除 Redis（第一次）
3. 500ms 后再次删除 Redis（处理并发读写导致的脏数据）

## 容量规划

| 组件 | 规格 | 说明 |
|------|------|------|
| L1 Caffeine | 单机 20MB | 缓存 1 万热点 SKU |
| Redis Cluster | 3 主 3 从，每主 16GB | 1000 万 × 2KB ≈ 20GB |
| 峰值 QPS 支撑 | 单 Redis 节点 10 万 QPS | 3 节点共 30 万，L1 拦截剩余 |
