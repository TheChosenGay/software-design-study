# 日志系统设计 - 参考答案

## 整体架构

```
客户端 SDK / Agent
       ↓
  Kafka（消息队列）
       ↓
  消费者集群（Flink/Logstash）
       ↙            ↘
Elasticsearch      对象存储（S3/OSS）
（热数据 7 天）      （冷数据 90 天）
```

## 各层设计

### 1. 采集层

**方案：轻量 SDK + 本地缓冲**
- 客户端 SDK 异步写入，本地内存队列缓冲（容量 10MB）
- 批量发送到 Kafka，batch size = 1000 条 或 100ms 触发一次
- 网络异常时本地磁盘持久化，恢复后重传

### 2. 消息队列（Kafka）

- Topic 按服务分区，partition 数 = 服务实例数 × 2
- 副本数 = 3，ISR 机制保证不丢数据
- 保留时间 24h 作为缓冲

### 3. 存储层

| 数据类型 | 存储 | 保留时长 | 说明 |
|---------|------|---------|------|
| 热数据 | Elasticsearch | 7 天 | 支持全文检索和聚合 |
| 温数据 | ClickHouse | 30 天 | 结构化查询，成本低 |
| 冷数据 | S3/OSS（Parquet） | 90 天 | 按需查询，极低成本 |

### 4. 索引设计

ES 索引按天分片：`logs-{service}-{YYYY.MM.DD}`
- 主 shard = 5，副本 = 1
- Mapping 关闭 `_source` 存储裸文本，减少存储

### 5. 查询层

- 简单查询：直接走 ES Query DSL
- 复杂分析：走 ClickHouse SQL
- 历史归档：Athena / Presto on S3

## 关键指标达成

| 指标 | 方案 | 预期 |
|------|------|------|
| 写入延迟 P99 | Kafka 异步 + 批量写 | < 5ms |
| 查询延迟 | ES 索引优化 | < 200ms |
| 可用性 | 多副本 + 消费者重试 | 99.9% |
| 存储成本 | 冷热分层 | 降低 70% |
