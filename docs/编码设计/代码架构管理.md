# 代码架构管理

> 核心目标：让变化的代价尽量小。稳定的东西（业务规则）不依赖易变的东西（技术细节）。

## 架构的本质：管理变化

代码里有两类东西：

- **稳定的**：核心业务规则（用户下单、库存扣减逻辑）
- **易变的**：技术细节（用什么数据库、UI 框架、推送服务）

好架构：稳定的东西不依赖易变的东西。现实中很多项目反过来——业务逻辑直接调用数据库 SDK、UI 框架，一换技术栈就要大改业务代码。

---

## 分层架构

把代码按职责分层，依赖只能单向向下：

```
┌─────────────────┐
│   UI / 表现层    │  处理用户交互，展示数据
├─────────────────┤
│   业务逻辑层     │  核心规则，不依赖任何技术细节
├─────────────────┤
│   数据访问层     │  与存储、网络打交道
└─────────────────┘
```

**依赖方向**：UI → 业务 → 数据，单向，不能反转。

### 违反分层的典型症状

```swift
// 症状 1：UI 层直接操作数据库
class UserViewController {
    func loadUser() {
        let result = db.query("SELECT * FROM users")  // UI 直接碰 DB
    }
}

// 症状 2：业务层依赖 UI 框架
class OrderService {
    func submit() {
        UIAlertView.show("提交成功")  // 业务层用了 UI 组件
    }
}

// 症状 3：业务规则写在 UI 层
class OrderViewController {
    func submitButtonTapped() {
        if total > 10000 {
            showAlert("单笔订单不能超过 10000 元")  // 业务规则，不该在这里
        }
    }
}
```

## 整洁架构（Clean Architecture）

分层架构的升级版。核心思想：**业务核心在最里层，技术细节在最外层，依赖只能由外向内**。

```
        ┌──────────────────────────┐
        │       框架和驱动          │  数据库、UI、网络（最易变）
        │  ┌────────────────────┐  │
        │  │    接口适配层       │  │  Controller、Presenter、Gateway
        │  │  ┌──────────────┐  │  │
        │  │  │  应用业务层   │  │  │  Use Cases / Service（用例）
        │  │  │  ┌────────┐  │  │  │
        │  │  │  │ 实体层  │  │  │  │  核心业务规则（最稳定）
        │  │  │  └────────┘  │  │  │
        │  │  └──────────────┘  │  │
        │  └────────────────────┘  │
        └──────────────────────────┘
```

**关键规则**：内层不知道外层的存在。业务核心完全不依赖数据库、UI、网络。

### 实际代码体现

```swift
// 实体层：纯业务对象，没有任何框架依赖
struct Order {
    let id: String
    let items: [OrderItem]
    var totalAmount: Double {
        items.reduce(0) { $0 + $1.price }
    }
    func validate() -> Bool { totalAmount > 0 && !items.isEmpty }
}

// 用例层：业务流程，依赖接口而非具体实现
class SubmitOrderUseCase {
    private let orderRepo: OrderRepository    // 接口
    private let paymentService: PaymentService  // 接口

    func execute(order: Order) -> Result {
        guard order.validate() else { return .failure(.invalidOrder) }
        orderRepo.save(order)
        paymentService.charge(order.totalAmount)
        return .success
    }
}

// 外层：具体实现，依赖内层定义的接口
class MySQLOrderRepository: OrderRepository { ... }
class StripePaymentService: PaymentService { ... }
```

内层（Order、SubmitOrderUseCase）完全不知道数据库是 MySQL 还是 PostgreSQL，支付是 Stripe 还是支付宝。

## 模块化：横向切分

分层是纵向的职责划分，模块化是横向的业务切分：

```
/OrderModule
  - Order.swift（实体）
  - SubmitOrderUseCase.swift（用例）
  - OrderRepository.swift（接口）
  - MySQLOrderRepository.swift（实现）

/UserModule
  - User.swift
  - RegisterUserUseCase.swift
  - UserRepository.swift

/NotificationModule
  - NotificationService.swift（接口）
  - APNsNotificationService.swift（实现）
```

**模块间通信原则**：通过接口通信，不直接依赖对方的具体类。

## Repository vs Service 命名规范

这是常见的命名混淆：

| 命名 | 职责 | 只做什么 |
|------|------|---------|
| `Repository` | 数据存取 | get / save / delete，不含业务逻辑 |
| `Service` / `UseCase` | 业务流程 | 调用 Repository，执行业务规则 |

```swift
// 正确划分
protocol OrderRepository {
    func get(id: String) -> Order?
    func save(order: Order)
}

class CartService {
    private let orderRepo: OrderRepository
    private let notifier: NotificationService

    func submitCart(userId: String) -> Result {
        // 业务逻辑在 Service 里
        let order = buildOrder(userId: userId)
        guard order.validate() else { return .failure }
        orderRepo.save(order)
        notifier.send(...)
        return .success
    }
}
```

## 架构决策的权衡

没有完美架构，只有适合当前阶段的架构：

| 阶段 | 推荐架构 | 原因 |
|------|---------|------|
| 早期 / 小项目 | 简单分层（MVC） | 快速迭代，过早分层是负担 |
| 中期 / 业务复杂 | 分层 + 模块化 | 业务增长，需要清晰边界 |
| 大型 / 多团队 | 整洁架构 | 多团队并行，必须解耦 |

**过度设计的陷阱**：5 个接口、3 层 Wrapper、一堆 UseCase，但整个项目只有 3 个页面。架构复杂度要匹配业务复杂度。

---

## 识别架构腐化的信号

| 信号 | 说明 |
|------|------|
| 改一个需求要动 5 个文件 | 模块边界不清晰，职责扩散 |
| 不敢改某段代码，怕牵连别的 | 隐式耦合太多 |
| 新人看代码不知道从哪入手 | 缺少清晰的分层和模块结构 |
| 单元测试很难写 | 业务和技术细节耦合，依赖难以替换 |
| 同样的逻辑散落在多处 | 缺少业务核心层，逻辑在各层重复 |

### 题目 1：识别架构问题

下面是一个 iOS 的 ViewController，找出所有架构层面的问题：

```swift
class OrderViewController: UIViewController {
    func submitButtonTapped() {
        let db = SQLiteDatabase()
        let order = db.query("SELECT * FROM cart WHERE userId = \(currentUserId)")

        if order.items.isEmpty {
            showAlert("购物车为空")
            return
        }

        let total = order.items.reduce(0) { $0 + $1.price }
        if total > 10000 {
            showAlert("单笔订单不能超过 10000 元")
            return
        }

        db.execute("INSERT INTO orders ...")

        let apns = APNsService()
        apns.send("您的订单已提交")

        navigationController?.pushViewController(OrderSuccessVC(), animated: true)
    }
}
```

<details>
<summary>查看答案与点评</summary>

**问题 1**：UI 层直接依赖数据层（SQLiteDatabase），跨层访问，违反分层原则。

**问题 2**：依赖具体实现（SQLiteDatabase、APNsService），无法替换，无法测试。

**问题 3**：业务规则写在 UI 层（购物车为空校验、10000 元限额）。换 UI 框架，业务规则要重写。

**问题 4**：SQL 字符串拼接，存在注入风险。

**问题 5**：ViewController 承担了查询、校验、存储、通知、导航五个职责，严重违反单一职责。

</details>

### 题目 2：分层重构

把题目 1 按分层架构重构，写出各层的核心代码结构。

<details>
<summary>查看答案与点评</summary>

```swift
// 实体层：纯业务对象
struct Order {
    let items: [OrderItem]
    var totalAmount: Double { items.reduce(0) { $0 + $1.price } }

    func validate() -> OrderValidationResult {
        if items.isEmpty { return .failure(.emptyCart) }
        if totalAmount > 10000 { return .failure(.exceedsLimit) }
        return .success
    }
}

// 数据层接口
protocol OrderRepository {
    func getCart(userId: String) -> Order
    func save(order: Order)
}

// 通知接口
protocol NotificationService {
    func send(_ message: String)
}

// 业务层：CartService 协调所有逻辑，提供回调
class CartService {
    private let orderRepo: OrderRepository
    private let notifier: NotificationService

    init(orderRepo: OrderRepository, notifier: NotificationService) { ... }

    func submitCart(userId: String, completion: (Result<Void, OrderError>) -> Void) {
        let order = orderRepo.getCart(userId: userId)
        switch order.validate() {
        case .failure(let error):
            completion(.failure(error))
        case .success:
            orderRepo.save(order)
            notifier.send("您的订单已提交")
            completion(.success(()))
        }
    }
}

// UI 层：只做展示决策，不含业务逻辑
class OrderViewController: UIViewController {
    private let cartService: CartService  // 构造注入

    func submitButtonTapped() {
        cartService.submitCart(userId: currentUserId) { [weak self] result in
            switch result {
            case .success:
                self?.navigationController?.pushViewController(OrderSuccessVC(), animated: true)
            case .failure(let error):
                self?.showAlert(error.localizedDescription)
            }
        }
    }
}
```

**关键点**：
- UI 层只决定"弹窗还是跳转"，不含任何业务规则
- 业务规则（校验逻辑）在 Order 实体和 CartService 里
- 所有依赖通过接口注入，便于替换和测试

</details>

### 题目 3：架构选型

你在做一个新的频道笔记功能模块，预计有：笔记列表、笔记详情、创建笔记、标签管理 4 个页面，后续可能接入 AI 摘要。你会选择什么架构？

<details>
<summary>查看答案与点评</summary>

**推荐：整洁架构**

```swift
// 实体层：核心业务对象，以后加 AI 摘要只需加字段
struct Book {
    let id: String
    let title: String
    let content: String
    var summary: String?      // AI 摘要后续加在这里
    let tags: [String]
    let createdAt: Date
}

// 数据层接口（Repository 只管存取）
protocol BookRepository {
    func getList(offset: Int, limit: Int) -> [Book]
    func get(id: String) -> Book?
    func create(book: Book)
}

// 用例层（Service 管业务流程）
class BookService {
    private let bookRepo: BookRepository
    private let aiService: AIService?   // 可选，后续接入

    func getList(offset: Int) -> [Book] { bookRepo.getList(offset: offset, limit: 20) }
    func getDetail(id: String) -> Book? { bookRepo.get(id: id) }
    func create(title: String, content: String, tags: [String]) { ... }

    // AI 摘要逻辑在 Service 里，不在 Repository
    func generateSummary(bookId: String) -> String? {
        guard let book = bookRepo.get(id: bookId),
              let ai = aiService else { return nil }
        let summary = ai.summarize(book.content)
        bookRepo.save(book.withSummary(summary))
        return summary
    }
}

// UI 层直接调用 BookService
class BookListViewController {
    private let bookService: BookService
    ...
}
```

**关键点**：
- AI 摘要的**调用逻辑**在 `BookService`（用例层），不是 `BookRepository`（数据层）
- AI 服务通过接口注入，换哪家 AI 不影响业务代码
- `Book` 结构体加字段只是数据变化，调用逻辑的扩展在 Service 里

</details>

## 薄弱点记录

### SQL 注入是高频遗漏点

**问题**：连续两课分析代码问题，都没有主动识别出 SQL 字符串拼接的注入风险。

**正确做法**：看到数据库查询代码，第一反应检查参数是否参数化。这应该成为条件反射。

### Repository vs Service 命名混用

**问题**：把数据存取接口命名为 `OrderRepoService`，混淆了两个职责。

**正确做法**：`Repository` 只管 get/save；`Service` / `UseCase` 管业务流程，可以调用 Repository。

### 新功能扩展的层次归属

**问题**：以为"加 AI 摘要只需要在 struct 里加字段"，忽略了调用逻辑要在用例层（Service）里扩展。

**正确做法**：struct 加字段是数据层面的变化；AI 摘要的触发、调用、存储逻辑属于用例层，写在 Service 里。
