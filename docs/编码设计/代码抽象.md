# 代码抽象

> 核心原则：抽象 = 隐藏细节，暴露意图。让调用方只知道"做什么"，不知道"怎么做"。

## 三个层次

### 1. 命名抽象（最基础）

函数名表达意图，不暴露技术实现细节：

```
// 差：暴露实现
func fetchUserDataFromRemoteAPIAndParseJSONAndSaveToCache(userId: String)

// 好：表达意图
func loadUser(userId: String)
```

**规律**：函数名过长是症状，单一职责才是病根。名字不得不列出所有步骤，说明函数做了太多事。

正确做法是拆分：

```
func fetchUser(userId: String) -> UserDTO
func parseUser(dto: UserDTO) -> User
func cacheUser(user: User)

// 上层协调，名字干净
func loadUser(userId: String) {
    let dto = fetchUser(userId)
    let user = parseUser(dto)
    cacheUser(user)
}
```

---

### 2. 接口抽象（核心）

用协议/接口隔离调用方和实现方，使实现可替换：

```
// 消息本身也应该结构化，不用裸字符串
struct Notification {
    title: String
    body: String
    userId: String
}

// 调用方只依赖接口
interface NotificationService {
    func send(notification: Notification) async -> Result
}

// 多种实现可以互换
class APNsNotificationService: NotificationService { ... }
class JPushNotificationService: NotificationService { ... }

// 调用方通过构造注入，完全不感知底层实现
class OrderService {
    init(notifier: NotificationService) { ... }
}
```

**关键判断**：换掉实现，调用方代码需不需要改？需要改 → 抽象不够；不需要改 → 抽象到位。

**常见坑**：用裸字符串/裸参数是弱抽象，不同场景有不同字段需求时扩展困难，应把数据结构化成对象。

---

### 3. 行为抽象（进阶）

把变化的行为提取出来作为参数，消除 if-else 随业务增长膨胀：

```
// 差：每增加一种类型就要改这里
func process(type) {
    if type == "A": doA()
    elif type == "B": doB()
}

// 好：行为本身成为参数（策略模式的本质）
func process(handler: Handler) {
    handler.execute()
}
```

## 抽象时机

### Rule of Three

- 第 1 次：直接写
- 第 2 次：忍住，记住它
- **第 3 次出现**：提取抽象

**过早抽象的代价**：为"可能的未来需求"设计了复杂接口，结果变化方向完全不同，接口反而成了负担。

### 另一个信号：改一处要改多处

修改一个逻辑需要同时改多个地方 → 这段逻辑存在隐式重复，没有被正确抽象。

---

## 抽象层次一致性

同一个函数里不能同时出现"业务语言"和"技术细节语言"：

```
// 差：混合层次
func submitOrder(order) {
    validateOrder(order)            // 业务层
    db.execute("INSERT INTO ...")   // 技术层，层次不对
    sendEmail(order.user.email)     // 业务层
}

// 好：统一在业务层
func submitOrder(order) {
    validateOrder(order)
    orderRepository.save(order)
    notificationService.notify(order)
}
```

## 常见坑速查

| 症状 | 根因 | 解法 |
|------|------|------|
| 函数名过长 | 函数职责过多 | 先拆函数，再起短名 |
| 换实现要改调用方 | 缺少接口抽象 | 引入协议/接口 |
| if-else 随需求膨胀 | 行为没有抽象 | 策略模式，行为作为参数 |
| 改一处要改多处 | 隐式重复 | Rule of Three，提取公共抽象 |
| 裸字符串传参 | 弱抽象 | 结构化成对象 |
| 抽象层次混乱 | 业务/技术混写 | 同一函数统一抽象层次 |

## 性能陷阱：DateFormatter

`DateFormatter` 创建成本高，每次调用都 new 一个会有性能问题（iOS 常见坑）。

```swift
// 差：每次都创建新实例
func formatDate(_ date: Date) -> String {
    let formatter = DateFormatter()
    formatter.dateFormat = "yyyy-MM-dd"
    return formatter.string(from: date)
}

// 好：缓存复用 + Extension 惯用法
extension Date {
    func formatted(_ format: String = "yyyy-MM-dd") -> String {
        DateFormatter.cached(format).string(from: self)
    }
}

extension DateFormatter {
    private static var cache: [String: DateFormatter] = [:]
    static func cached(_ format: String) -> DateFormatter {
        if let f = cache[format] { return f }
        let f = DateFormatter()
        f.dateFormat = format
        cache[format] = f
        return f
    }
}

// 调用
date.formatted()          // "2026-04-20"
date.formatted("MM/dd")   // "04/20"
```

### 题目 2：接口抽象

你有一个消息推送模块，目前直接调用 APNs 发送通知。产品说未来可能接入第三方推送（极光、个推）。你会怎么设计这个模块的抽象？

<details>
<summary>查看答案与点评</summary>

**关键点**：消息本身也要结构化，不能用裸字符串。

```
// 消息结构化

    title: String
    body: String
    userId: String
}

// 接口定义
interface NotificationService {
    func send(notification: Notification) async -> Result
}

// 具体实现
class APNsNotificationService: NotificationService { ... }
class JPushNotificationService: NotificationService { ... }

// 调用方构造注入
class OrderService {

}
```

**常见错误**：用 `msg: String` 裸字符串，不同平台需要 title、body、target 等多个字段时无法扩展。

### 题目 3：识别抽象时机

下面三段代码出现在同一个项目的不同地方，你会怎么处理？

```
// 地方 A
let dateStr = DateFormatter().string(from: date, format: "yyyy-MM-dd")

// 地方 B
let dateStr = DateFormatter().string(from: createdAt, format: "yyyy-MM-dd")

// 地方 C
let dateStr = DateFormatter().string(from: updatedAt, format: "yyyy-MM-dd")
```

<details>
<summary>查看答案与点评</summary>

**触发 Rule of Three**：同样的逻辑出现了 3 次，应该提取抽象。

**注意性能陷阱**：`DateFormatter` 创建成本高，提取函数时要同时处理缓存：

```swift
extension Date {
    func formatted(_ format: String = "yyyy-MM-dd") -> String {
        DateFormatter.cached(format).string(from: self)
    }
}

extension DateFormatter {
    private static var cache: [String: DateFormatter] = [:]
    static func cached(_ format: String) -> DateFormatter {
        if let f = cache[format] { return f }
        let f = DateFormatter()
        f.dateFormat = format
        cache[format] = f
        return f
    }
}
```

用 Extension 挂在 `Date` 上比独立函数更符合语言惯用法。

</details>

## 薄弱点记录

### 消息结构化意识

**问题**：设计推送接口时用裸字符串 `msg: String`，没有考虑不同平台需要 title、body、target 等多个字段。

**正确做法**：消息本身也要抽象成结构体，接口签名用 `Notification` 对象而不是裸字符串。

**正确做法**：用字典缓存不同 format 对应的 DateFormatter 实例，避免重复创建。
