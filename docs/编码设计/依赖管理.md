# 依赖管理

> 核心原则：依赖抽象，不依赖具体实现。依赖从外部注入，不在内部创建。

## 依赖的两种形态

### 硬依赖（危险）

A 直接创建或引用 B 的具体类型：

```
class OrderService {
    let db = MySQLDatabase()   // 直接 new，换不掉
}
```

后果：想换实现要改调用方；写单元测试必须启动真实数据库；B 出问题 A 直接崩。

### 软依赖（安全）

A 只依赖 B 的抽象，具体实现从外面传入：

```
class OrderService {
    let db: Database
    init(db: Database) { ... }
}
```

---

## 依赖倒置（DIP）

> 高层模块不应该依赖低层模块，两者都应该依赖抽象。

直觉设计（高层依赖低层）：
```
OrderService → MySQLDatabase
```

依赖倒置后：
```
OrderService → Database（接口）← MySQLDatabase
```

**"倒置"的含义**：原本高层依赖低层，倒置后低层反过来依赖高层定义的抽象，箭头方向改变了。

---

## 依赖注入（DI）

依赖倒置是原则，依赖注入是落地方式。核心：**不在类内部创建依赖，从外部传入**。

### 三种注入方式

**1. 构造注入（推荐）** — 整个生命周期都需要的依赖

```
class ReportGenerator {
    private let db: Database
    private let uploader: Uploader

    init(db: Database, uploader: Uploader) {
        self.db = db
        self.uploader = uploader
    }

    func generate(userId: String) {
        let user = db.getUser(userId: userId)
        // 渲染 PDF...
        uploader.upload(pdf.output())
    }
}

// 组装一次，之后直接用
let generator = ReportGenerator(
    db: MySQLDatabase(host: "prod-db.internal"),
    uploader: AWSS3Uploader(bucket: "reports")
)
generator.generate(userId: "123")
generator.generate(userId: "456")
```

**2. 方法注入** — 只在某个操作临时需要的依赖

```
func generate(userId: String, db: Database, uploader: Uploader) { ... }
```

注意：不要把长期依赖用方法注入，否则每次调用都要传，负担转移给调用方。

**3. 属性注入** — 可选依赖，允许后续设置

```
class OrderService {
    var logger: Logger?
}
```

### 选择依据

| 依赖类型 | 推荐方式 |
|---------|---------|
| 对象整个生命周期都需要 | 构造注入 |
| 只在某个方法调用时需要 | 方法注入 |
| 可选、允许为空 | 属性注入 |

## 循环依赖

### 问题

```
UserService → NotificationService（需要发通知）
NotificationService → UserService（需要查用户偏好）
```

后果：编译顺序无法确定；改一个地方另一个意外崩；无法独立测试任何一个模块。

### 解法：提取接口到独立的 Protocols 模块

```
// ---- Protocols 模块：只有接口，不依赖任何人 ----
interface UserServiceProtocol {
    func getUserPreference(userId: String) -> UserPreference
}

interface NotificationServiceProtocol {
    func send(notification: Notification)
}

// ---- UserService：依赖 Notification 的接口，不依赖具体类 ----
class UserService: UserServiceProtocol {
    private let notifier: NotificationServiceProtocol
    init(notifier: NotificationServiceProtocol) { ... }
}

// ---- NotificationService：依赖 User 的接口，不依赖具体类 ----
class NotificationService: NotificationServiceProtocol {
    private let userService: UserServiceProtocol
    init(userService: UserServiceProtocol) { ... }
}
```

依赖方向：
```
UserService         → Protocols 模块
NotificationService → Protocols 模块
两者不再直接依赖，循环消失
```

**命名提示**：放接口的模块不要叫 `util`，应叫 `Protocols` / `Contracts` / `Interfaces`，体现其契约性质。

## 接口隔离（ISP）

不要强迫调用方依赖它不需要的方法，接口按需拆细：

```
// 差：一个大接口，Robot 被迫实现 eat/sleep
interface Worker {
    func work()
    func eat()
    func sleep()
}

// 好：拆细，按需组合
interface Workable  { func work() }
interface Eatable   { func eat() }
interface Sleepable { func sleep() }

class Robot: Workable { func work() { ... } }
class Human: Workable, Eatable, Sleepable { ... }
```

---

## 安全意识：SQL 注入

依赖数据库时，查询参数不能用字符串拼接：

```
// 危险：userId 可以是 "1; DROP TABLE users"
db.query("SELECT * FROM users WHERE id = \(userId)")

// 安全：参数化查询
db.query("SELECT * FROM users WHERE id = ?", params: [userId])
```

**原则**：外部输入永远不可信，不能直接拼进 SQL / 命令行 / HTML。

## 常见坑速查

| 症状 | 根因 | 解法 |
|------|------|------|
| 换实现要改调用方 | 依赖具体类 | 依赖倒置，改依赖接口 |
| 写测试要启动真实数据库 | 依赖在内部创建 | 构造注入，测试时传 Mock |
| 每次调用都要传依赖 | 长期依赖用了方法注入 | 改构造注入 |
| 两个模块互相依赖 | 循环依赖 | 提取接口到 Protocols 模块 |
| 实现空方法应付接口 | 接口过胖 | 接口隔离，拆细按需依赖 |

## 练习题

### 题目 1：识别依赖问题

下面的代码有哪些依赖问题？列出来并说明原因：

```
class ReportGenerator {
    func generate(userId: String) {
        let db = MySQLDatabase(host: "prod-db.internal")
        let user = db.query("SELECT * FROM users WHERE id = \(userId)")
        let pdf = PDFRenderer()
        pdf.render(user)
        let uploader = AWSS3Uploader(bucket: "reports")
        uploader.upload(pdf.output())
    }
}
```

<details>
<summary>查看答案与点评</summary>

**问题 1**：依赖具体实现（MySQLDatabase、AWSS3Uploader、PDFRenderer），无法替换，无法测试。

**问题 2**：依赖在函数内部创建，无法从外部注入 Mock。

**问题 3**：SQL 字符串拼接，存在注入风险。应用参数化查询：
```
db.query("SELECT * FROM users WHERE id = ?", params: [userId])
```

**问题 4**：函数职责过多（查用户、渲染 PDF、上传），应拆分。

</details>

### 题目 2：依赖注入改造

把题目 1 的代码用依赖注入重构，写出接口 + 实现 + 注入方式。

<details>
<summary>查看答案与点评</summary>

```
// 接口定义
interface Database {
    func getUser(userId: String) -> User
}

interface Uploader {
    func upload(data: Data) -> Result
}

// 构造注入（推荐，整个生命周期都需要这两个依赖）
class ReportGenerator {
    private let db: Database
    private let uploader: Uploader

    init(db: Database, uploader: Uploader) {
        self.db = db
        self.uploader = uploader
    }

    func generate(userId: String) {
        let user = db.getUser(userId: userId)
        // 渲染 PDF...
        uploader.upload(pdf.output())
    }
}

// 具体实现
class MySQLDatabase: Database { ... }
class AWSS3Uploader: Uploader { ... }

// 调用方组装一次
let generator = ReportGenerator(
    db: MySQLDatabase(host: "prod-db.internal"),
    uploader: AWSS3Uploader(bucket: "reports")
)
generator.generate(userId: "123")
```

**关键点**：用构造注入而非方法注入——db 和 uploader 是整个生命周期都需要的依赖，不应每次调用都传。

### 题目 3：打破循环依赖

`UserService` 需要发送通知，依赖 `NotificationService`。
`NotificationService` 发通知时需要查用户偏好，依赖 `UserService`。
这是循环依赖，如何打破？

<details>
<summary>查看答案与点评</summary>

提取接口到独立的 Protocols 模块，两个类都只依赖接口，直接循环消失：

```
// Protocols 模块（不依赖任何人）
interface UserServiceProtocol {
    func getUserPreference(userId: String) -> UserPreference
}

interface NotificationServiceProtocol {
    func send(notification: Notification)
}

// 各自依赖对方的接口，不依赖具体类
class UserService: UserServiceProtocol {
    private let notifier: NotificationServiceProtocol
    init(notifier: NotificationServiceProtocol) { ... }
}

class NotificationService: NotificationServiceProtocol {
    private let userService: UserServiceProtocol
    init(userService: UserServiceProtocol) { ... }
}
```

**命名提示**：放接口的模块叫 `Protocols` / `Contracts`，不要叫 `util`。

</details>
